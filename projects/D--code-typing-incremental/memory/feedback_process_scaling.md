Don't run the full brainstorming-skill flow (design doc → self-review → user review → writing-plans) on atomic decisions like naming, config values, or single-choice picks. The skill's own guidance allows "a few sentences for truly simple projects" — use that license. A name pick belongs in a short note, not a 10-section spec with trademark class references.

**Why:** User pushed back on a 150-line design spec for *"pick a project name"* — "why are you writing a design spec for a name decision?" Process applied for its own sake produces noise, not signal, and erodes trust that I'm calibrating effort to value.

**How to apply:** If the decision is a single choice with no sub-decisions, capture the outcome + 2-3 sentence rationale + availability/evidence inline or in a short notes file. Skip the design-doc-commit-review-writing-plans sequence. Reserve that flow for actual multi-step builds where the decomposition itself has value.
